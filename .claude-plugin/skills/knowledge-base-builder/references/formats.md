# Format Examples

Adapt structure to content type. These are starting points, not rigid templates.

## Summary Formats

### Pattern/Concept Summary

```markdown
# [Title]

## Overview
[2-3 sentences: what it is, why it matters]

## Key Concepts
- [Core idea 1]
- [Core idea 2]

## How It Works
[Main content - adapt section name]

## When to Use
- [Use case 1]
- [Use case 2]

## Takeaways
- [Key point 1]
- [Key point 2]
```

### API/Reference Summary

```markdown
# [API/Module Name]

## Purpose
[What this does]

## Key Methods/Endpoints

| Method | Description |
|--------|-------------|
| `func1()` | Does X |
| `func2()` | Does Y |

## Example
[Minimal working example]

## Notes
- [Important caveat]
```

### Process/Tutorial Summary

```markdown
# [Process Name]

## Goal
[What you'll accomplish]

## Steps
1. [Step 1]
2. [Step 2]
3. [Step 3]

## Common Issues
- [Issue]: [Solution]
```

## README Structure

Core elements (adapt as needed):

```markdown
# [Knowledge Base Name]

[One-line description]. Source: [origin].

## Structure

\`\`\`
knowledge-base/
├── README.md
├── [category]/
└── [category]/
\`\`\`

## Contents

| File | Description |
|------|-------------|
| [link](path) | Brief description |

## Navigation Guide
[How to find what you need]

## Changelog

| Version | Date | Changes |
|---------|------|---------|
| 1.0 | YYYY-MM-DD | Initial |
```

## Flexibility Guidelines

**Formal content** (textbooks, specs):
- Use metadata headers
- Consistent section structure
- Numbered files

**Informal content** (notes, guides):
- Skip metadata if not useful
- Adapt sections to content
- Descriptive filenames

**Mixed content**:
- Consistency within categories
- Different formats across categories OK
