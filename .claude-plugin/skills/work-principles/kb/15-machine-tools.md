## Key Bridgewater Tools

### 1. Dot Collector
**Purpose**: Real-time meeting feedback

**How it works**:
```
During meeting → Rate people on attributes
Over time → Patterns emerge
Weighted by believability → Meaningful signal
```

**Benefits**:
- Surfaces hidden disagreements
- Tracks patterns over time
- Makes feedback normal

### 2. Baseball Cards
**Purpose**: Track person attributes

**Contains**:
- Strengths/weaknesses
- Believability by domain
- Historical performance
- Development areas

**Use**:
- Role matching
- Team composition
- Believability weighting

### 3. Issue Log
**Purpose**: Mandatory problem recording

**Rules**:
- Log all problems
- Include severity
- Assign responsibility
- Track resolution

### 4. Pain Button
**Purpose**: Capture painful moments

**Process**:
```
Pain occurs → Record immediately → Reflect → Extract lesson → Share
```

## Protocol Design

### Good Protocol Qualities
| Quality | Description |
|---------|-------------|
| Clear | Unambiguous steps |
| Complete | All cases covered |
| Efficient | Minimal unnecessary steps |
| Enforced | Can't easily skip |
| Measured | Compliance tracked |

### Building Protocols
1. Identify repeating situation
2. Define best practice
3. Document steps
4. Build into tools
5. Measure compliance
6. Iterate based on results

## Tool Adoption Challenges

### Common Resistance
| Resistance | Response |
|------------|----------|
| "Takes too much time" | Show long-term benefit |
| "I don't need it" | Require it anyway |
| "It's not flexible" | Improve the tool |
| "I forget to use it" | Automate triggers |

### Adoption Strategy
- Start with believers
- Show early wins
- Make it easier than alternative
- Require for critical processes
